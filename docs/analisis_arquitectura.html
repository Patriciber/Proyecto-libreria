Documentación Técnica: Proyecto "Lumière"

Análisis Estructural y Diseño de Software Orientado a Dominio

1. Resumen Ejecutivo

El proyecto "Lumière" implementa un diseño de software basado en la Arquitectura Hexagonal (también conocida como arquitectura de Puertos y Adaptadores). Este paradigma arquitectónico coloca la lógica de negocio (el Dominio) en el núcleo absoluto del sistema, aislándola de las tecnologías de infraestructura, bases de datos y capas de presentación.

La principal ventaja de este enfoque es que el núcleo de la aplicación es completamente agnóstico al mundo exterior. Esto significa que nuestra lógica de librería (búsqueda de libros, validación de inventario, registro de usuarios) no conoce la existencia del marco web HTTP (FastAPI) ni el medio de persistencia (archivos JSON o una base de datos SQL). Toda comunicación hacia y desde el exterior se realiza a través de interfaces bien definidas o "Puertos".

3. Justificación de Refactorización

La transición de una arquitectura monolítica (y acoplada) tradicional, donde los controladores manejan tanto la lógica de red como el acceso a datos, hacia esta estructura Hexagonal aporta beneficios formidables:

  • Incremento Crítico en la Testabilidad: En un monolito, probar una función de "registro de usuario" requería levantar una base de datos real y un servidor HTTP. Gracias a la inyección de dependencias y los "Puertos" (interfaces), podemos inyectar un MockUserAdapter o un book_repository_mock.py durante los tests para verificar la lógica de negocio en milisegundos mediante pytest, sin afectar el estado real.
  • Modularidad Inflexible: Si el día de mañana se decide migrar el almacenamiento de los libros de un archivo estático JSON a una base de datos en la nube (ej. AWS RDS / PostgreSQL), no se debe modificar ni una sola línea de código en LibraryService ni en APIRouters. Solo es necesario programar una nueva clase que implemente IBookRepository y declararla en el contenedor de dependencias (container.py).
  • Retención de Responsabilidad Única (SRP): Se ha asegurado que el routing (FastAPI) solo se encargue de serializar peticiones JSON y devolver respuestas HTTP, delegando toda la complejidad matemática y de negocio a los Servicios, cumpliendo el principio de Diseño Resiliente.

4. Detalle de Componentes

A continuación, desglosamos las responsabilidades estandarizadas de cada carpeta, archivo y módulo presente tanto en el lado del cliente (Frontend) como en el servidor (Backend).

🖥️ Front-end (Capa de Presentación)

Aplicación de tipo SPA (Single Page Application) construida sin frameworks reactivos masivos, favoreciendo el uso de Vanilla JavaScript, HTML5 y CSS3.

  • index.html: Estructura semántica de la página. Define los contenedores donde se inyectará dinámicamente el catálogo de libros y el carrito.
  • styles.css: Sistema de estilos moderno y responsive. Gestiona el diseño visual general, las tarjetas de los libros y la adaptabilidad a dispositivos móviles.
  • script.js: Centraliza la interactividad. Utiliza fetch() de forma asíncrona para consultar el catálogo al Backend, procesa el JSON resultante y manipula el DOM (`document.createElement`) para renderizar el catálogo y actualizar el estado del carrito en memoria del navegador.

⚙️ Back-end (Core & Infraestructura)

Construido sobre Python utilizando FastAPI como adaptador web principal. Organizado bajo estricta separación de carpetas por capas.

  • app/domain/ (Dominio): Contiene el corazón de la app. Los models/ (entidades y esquemas de datos como User y Book) y los servicios (reglas de negocio centrales sin dependencias externas).
  • app/ports/ (Interfaces): Definición de clases abstractas que dictan "qué" métodos debe implementar infraestructura, sin decir "cómo".
  • app/routers/ (Adaptador Primario): Controladores de red. Proveen los endpoints HTTP, reciben datos, llaman a los servicios de dominio, e interceptan y empaquetan las respuestas (FastAPI).
  • app/adapters/ (Adaptador Secundario): Capa tecnológica donde se implementan los Ports. Contiene la lógica de conexión al books.json mediante I/O o Mocks transitorios orientados a pruebas.
  • app/container.py: Archivo vital de ensamblaje (Dependency Injection). Centraliza la construcción de los adaptadores y se los provee a los servicios.
